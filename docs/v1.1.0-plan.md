## Feature 1 — Pipeline Setup Wizard

### Goal
A guided multi-step form for creating a new pipeline without writing YAML manually.
Replaces the blank code editor for users unfamiliar with the YAML schema.

### User Flow
```
Step 1: Basic info    — name, description, schedule type + interval/cron
Step 2: Source        — pick source type, fill required fields (host, path, topic …)
Step 3: Transforms    — add zero or more transforms, ordered list with up/down buttons
Step 4: Sinks         — add one or more sinks; per-sink serializer_out override
Step 5: Review        — rendered YAML preview + dry-run validation button
        [Save & Register]  →  POST /api/pipelines  (same as editor)
```

### UI
- New page: `tram-ui/src/pages/wizard.html` + `wizard.js`
- Nav link "New Pipeline" opens wizard by default; editor link remains for advanced users
- Step indicator bar at top (1–5), Back/Next buttons, keyboard-navigable
- Step 5 calls `POST /api/pipelines/dry-run` before enabling Save; shows pass/fail inline

### Backend
- No new API endpoints needed — wizard POSTs to existing `POST /api/pipelines`
- YAML is assembled client-side from wizard form state
- Field metadata (required fields, types, defaults) sourced from `GET /api/plugins`
  which already returns all registered connector keys by category

### YAML generation
```javascript
// wizard.js assembles PipelineConfig-compatible YAML from form state:
function buildYaml(state) {
  // schedule block, source block, transforms[], sinks[]
  // uses js-yaml or simple template strings
}
```

### Dependencies
- `js-yaml` (already a devDependency candidate) or a minimal YAML serialiser
- `GET /api/plugins` for dynamic connector type lists (already exists)

## Feature 2 — Live Metrics on Dashboard

### Goal
Replace static "last run" counters on the dashboard with auto-refreshing throughput,
error rate, and record flow metrics — without requiring Prometheus.

### Metrics exposed
| Metric | Source | Refresh |
|--------|--------|---------|
| Records/min (global) | DB `run_history` aggregation | 10 s |
| Records/min per pipeline | DB aggregation | 10 s |
| Error rate (last 15 min) | DB `run_history` WHERE status='error' | 10 s |
| Active pipelines (running/scheduled) | `manager.list_all()` | 10 s |
| Total runs today | DB count | 30 s |
| Avg run duration (last hour) | DB aggregation | 30 s |

### New API endpoint
```
GET /api/stats
```
Response:
```json
{
  "pipelines_total": 5,
  "pipelines_running": 1,
  "pipelines_scheduled": 3,
  "runs_today": 147,
  "runs_last_hour": 12,
  "records_in_last_15m": 48200,
  "records_out_last_15m": 48198,
  "errors_last_15m": 2,
  "avg_duration_last_hour_s": 1.4,
  "per_pipeline": [
    {"name": "sample-health", "runs_last_hour": 12, "records_in": 48200, "errors": 2}
  ]
}
```

### Backend
- New file: `tram/api/routers/stats.py`
- All queries against existing `run_history` table — no schema changes
- In-memory fallback (no DB): scan `manager.list_all()` run_history deques

### UI
- `dashboard.js`: replace static summary cards with polling `GET /api/stats` every 10 s
- Add sparkline mini-charts (records/min per pipeline, last 12 data points) using
  Canvas API directly — no charting library dependency
- "Live" indicator dot pulses green while polling is active

## Feature 3 — Alert Rules UI

### Goal
View, create, edit, and delete alert rules for a pipeline from the UI.
The backend engine (`AlertEvaluator`) and model (`AlertRuleConfig`) already exist —
this is purely surfacing them.

### Current state
- `AlertRuleConfig` model: `condition` (simpleeval expression), `action` (webhook/email),
  `webhook_url`/`email_to`, `subject`, `cooldown_seconds`
- `AlertEvaluator.check()` fires on every `manager.record_run()`
- Alerts are defined inline in pipeline YAML under `alerts:`
- No API endpoint to read/write alert rules independently of full pipeline YAML

### New API endpoints
```
GET    /api/pipelines/{name}/alerts          # list alert rules for a pipeline
POST   /api/pipelines/{name}/alerts          # add an alert rule
PUT    /api/pipelines/{name}/alerts/{idx}    # replace alert rule at index
DELETE /api/pipelines/{name}/alerts/{idx}    # remove alert rule at index
```
Implementation: read current YAML → mutate `alerts` list → re-save via `PUT /api/pipelines/{name}`
(a thin wrapper, no new DB tables)

### UI
- New "Alerts" tab on the pipeline detail page (`detail.html` / `detail.js`)
- Table: rule name | condition | action | cooldown | last fired | [Edit] [Delete]
- "Add Rule" modal with fields: name, condition (with variable reference tooltip),
  action type toggle (webhook / email), target URL or address, cooldown
- Available condition variables shown inline:
  `records_in`, `records_out`, `duration_s`, `error_count`, `status`

### Condition expression examples shown in UI
```
records_out == 0
error_count > 0
duration_s > 30
status == 'error'
```

## Feature 4 — YAML Diff Viewer (Versions Tab)

### Goal
When viewing pipeline version history, show a side-by-side (or unified) diff between
any two versions — or between a historical version and the currently active one.

### Current state
- `GET /api/pipelines/{name}/versions` returns list of `{version, created_at, is_active}`
- `GET /api/pipelines/{name}/versions/{v}` returns raw YAML text
- Versions tab in `detail.js` lists versions with a "Restore" button — no diff

### Changes

**UI (`detail.js`, `detail.html`)**
- Add "Diff" button next to each version row → opens diff modal
- Default: diff selected version vs. currently active version
- Optional: "Compare two versions" — two dropdowns, one for each side
- Diff rendered with a lightweight inline diff library or manual line-by-line comparison:
  - Added lines: green background
  - Removed lines: red background
  - Unchanged lines: dimmed
- No new backend API needed — fetch both YAML texts via existing version endpoint,
  compute diff client-side

**Client-side diff algorithm**
```javascript
// detail.js: Myers diff on split('\n') arrays
// Renders into a <pre> with colored spans — ~60 lines, no dependency
```

## Feature 5 — Connector Connectivity Test

### Goal
Before saving/deploying a pipeline, allow the user to test that a source or sink
connector can actually reach its target (TCP reachability, auth check, topic exists, etc.).

### New API endpoint
```
POST /api/connectors/test
Body: { "type": "kafka", "config": { "brokers": ["localhost:9092"], "topic": "pm-raw" } }
Response: { "ok": true, "latency_ms": 12, "detail": "Connected to 1/1 brokers" }
         { "ok": false, "error": "Connection refused: localhost:9092" }
```

### Backend
- New file: `tram/api/routers/connectors.py`
- Connector test implementations per plugin type — a new optional `test()` method on
  source/sink plugins, with a fallback TCP-connect probe for connectors that don't implement it

**Initial coverage (v1.1.0)**
| Connector | Test method |
|-----------|-------------|
| `kafka` | `KafkaProducer` bootstrap connect, list topics |
| `rest` | HTTP GET/HEAD to base URL, check status < 500 |
| `sql` | `engine.connect()` / `SELECT 1` |
| `local` | `os.path.exists()` on path |
| `sftp` | SSH TCP connect + auth handshake |
| `s3` / `gcs` / `azure_blob` | List bucket/container (1 object) |
| `mqtt` | TCP connect to broker host:port |
| `influxdb` | `GET /ping` health endpoint |
| `opensearch` / `elasticsearch` | `GET /` cluster info |
| others | TCP connect probe to host:port if extractable |

### UI
- Connectivity test button in the wizard (Step 2 Source, Step 4 Sinks)
- Also available in the editor toolbar: "Test Source" / "Test Sinks"
- Inline result: green "Reachable (12ms)" or red "Unreachable: Connection refused"

### Plugin interface addition
```python
# tram/core/base.py (or registry)
class ConnectorTestMixin:
    def test_connection(self) -> dict:
        """Returns {"ok": bool, "detail": str, "latency_ms": int}"""
        raise NotImplementedError
```

## Feature 6 — Pipeline Templates Library

### Goal
A browsable gallery of pre-built pipeline templates that can be deployed with one click.
TRAM already ships 20+ templates in `pipelines/` — this just surfaces them in the UI.

### Current state
- `pipelines/` directory has 20+ production-ready YAML templates:
  kafka-to-opensearch, sftp-pm-to-kafka, snmp-trap-receiver, syslog-to-opensearch,
  rest-nms-to-sql, webhook-alarm-fanout, etc.
- No API or UI to browse/deploy them

### New API endpoint
```
GET /api/templates
Response: [
  {
    "id": "kafka-to-opensearch",
    "name": "Kafka → OpenSearch",
    "description": "Consume JSON records from Kafka and index into OpenSearch",
    "tags": ["kafka", "opensearch", "streaming"],
    "source_type": "kafka",
    "sink_types": ["opensearch"],
    "schedule_type": "stream",
    "yaml": "..."
  },
  ...
]
```

### Backend
- New file: `tram/api/routers/templates.py`
- Reads `pipelines/` directory at startup (or on each request with 60 s cache)
- Parses each YAML to extract `name`, `source.type`, `sinks[].type`, `schedule.type`
- Description and tags derived from filename + parsed fields; or from an optional
  `# description:` / `# tags:` comment header in each YAML file

### UI
- New page: `tram-ui/src/pages/templates.html` + `templates.js`
- Nav section "Templates" (below Pipelines)
- Card grid layout — each card shows source→sink icon flow, name, tags, schedule badge
- Filter bar: by source type, sink type, schedule type
- "Deploy" button → opens editor with YAML pre-filled; user edits name + connection
  details, then saves
- "Preview" button → shows YAML in a read-only code block

### Template YAML header convention
```yaml
# description: Consume JSON records from Kafka, index into OpenSearch
# tags: kafka, opensearch, streaming
pipeline:
  name: kafka-to-opensearch
  ...
```
Parser reads first `# description:` and `# tags:` comment lines.

## Feature 7 — Pipeline Import / Export

### Goal
Download a pipeline's current YAML and re-upload it (to the same or another TRAM instance).
Enables config backup, GitOps-lite workflows, and cross-environment promotion.

### Export
**UI**: "Export YAML" button on pipeline detail page
**Mechanism**: `GET /api/pipelines/{name}` already returns YAML in `detail.to_detail_dict()["yaml"]`
No new endpoint needed — JS triggers a file download from the existing detail API.

```javascript
// detail.js
function exportYaml(name, yaml) {
  const blob = new Blob([yaml], {type: 'text/yaml'})
  const a = document.createElement('a')
  a.href = URL.createObjectURL(blob)
  a.download = `${name}.yaml`
  a.click()
}
```

### Import
**UI**: "Import YAML" button on Pipelines list page — file picker or drag-and-drop
**Mechanism**: reads file → calls existing `POST /api/pipelines` with YAML body
No new endpoint needed.

```javascript
// pipelines.js
async function importYaml(file) {
  const yaml = await file.text()
  await api.pipelines.create(yaml)   // POST /api/pipelines (already exists)
  await loadPipelines()
  toast(`Imported ${file.name}`)
}
```

### Version-aware re-import
If a pipeline with the same name already exists, prompt:
- "Replace" → `PUT /api/pipelines/{name}` (already exists, saves new version)
- "Rename" → edit name field before importing
- "Cancel"

No new backend code — reuses existing create/update endpoints.

## Implementation Order

Recommended sequence based on dependencies and user impact:

```
1. Templates Library      — pure read path, no risk, immediate value
2. Import / Export        — reuses existing API, minimal code
3. YAML Diff Viewer       — client-side only, no backend
4. Live Metrics Dashboard — new /api/stats endpoint + dashboard polling
5. Alert Rules UI         — new alert sub-endpoints + detail tab
6. Connectivity Test      — new /api/connectors/test + plugin interface
7. Wizard                 — most complex, builds on 1+6
```

Items 1–3 can be done in parallel (independent UI features, no shared state).
Items 4–7 are sequential by dependency.
