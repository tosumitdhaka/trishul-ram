 TRAM — Trishul Real-time Adapter & Mapper

 Context

 Build a lightweight, container-native Python daemon + CLI that moves and transforms telecom data (PM/FM/Logs) across
 protocols. Named TRAM as a standalone module, designed for future integration into the Trishul platform. v1 supports SFTP
 read/write; plugin architecture allows Kafka, OpenSearch, REST, SNMP, VES to be added with minimal effort. Data flows
 through a YAML-defined pipeline: Source → Deserialize → Transform chain → Serialize → Sink.

 The app always runs as a daemon. Pipelines are registered/deregistered/started/stopped at runtime via CLI or REST.
 Scheduling is per-pipeline and optional — batch pipelines poll on interval/cron, stream pipelines run continuously.

   serializer_in:
     type: csv
     delimiter: ","
     has_header: true

   transforms:
     - type: rename
       fields:
         ne_id: network_element_id
         ts:    timestamp

     - type: cast
       fields:
         rx_bytes:  int
         timestamp: datetime           # ISO-8601 → Python datetime
         active:    bool               # "true"/"1"/"yes" → True

     - type: add_field                 # expressions use simpleeval (safe sandbox)
       fields:
         rx_mbps:   "rx_bytes / 1_000_000"
         load_pct:  "round(rx_mbps / 1000 * 100, 2)"

     - type: drop
       fields: [rx_bytes, debug_flag]

     - type: value_map
       field: severity
       mapping:
         "1": CRITICAL
         "2": MAJOR
         "3": MINOR
       default: UNKNOWN

     - type: filter
       condition: "rx_mbps > 0"

   serializer_out:
     type: json
     indent: 2

   sink:
     type: sftp
     host: ${MED_SFTP_HOST}
     port: ${MED_SFTP_PORT:-22}
     username: ${MED_SFTP_USER}
     password: ${MED_SFTP_PASS}
     remote_path: /ingest/pm/
     filename_template: "pm_{pipeline}_{timestamp}.json"  # tokens: {pipeline} {timestamp} {source_filename}

   on_error: continue            # "continue" | "abort" | "retry"
   retry_count: 3
   retry_delay_seconds: 10

 Batch vs Stream Execution Modes

 The key architectural distinction. PipelineExecutor has two execution paths, selected by schedule.type:

 Batch Mode (interval, cron, manual)

 source.read() yields a finite set → process all → return RunResult → sleep → repeat
 - Driven by APScheduler (interval/cron) or REST/CLI trigger (manual)
 - Each run is a discrete unit with a RunResult, run_id, timestamps
 - Natural for SFTP file polling, REST API pulls

 Stream Mode (stream)

 source.read() is an infinite generator → process each record as it arrives → never returns
 - Runs in a dedicated thread per pipeline (not APScheduler)
 - No RunResult per-run — instead emits continuous metrics (records/sec, errors)
 - Natural for Kafka, NATS, gRPC streams, SNMP traps
 - Stopped only by pipeline stop or daemon shutdown

 # executor.py — conceptual

 class PipelineExecutor:
     def batch_run(self, config) -> RunResult:
         """Finite execution: reads all source files, transforms, writes, returns."""
         for raw, meta in source.read():          # source returns finite iterator
             records = ser_in.parse(raw)
             for t in transforms: records = t.apply(records)
             sink.write(ser_out.serialize(records), meta)
         return RunResult(...)

     def stream_run(self, config, stop_event: threading.Event):
         """Infinite execution: runs until stop_event is set."""
         for raw, meta in source.read():          # source blocks waiting for next message
             if stop_event.is_set(): break
             records = ser_in.parse(raw)
             for t in transforms: records = t.apply(records)
             sink.write(ser_out.serialize(records), meta)

 Plugin System — How to Add a New Connector

 Adding Kafka, OpenSearch, REST, SNMP, VES, etc. requires only:

 Step 1 — Create connector file(s)

 # tram/connectors/kafka/source.py
 from tram.registry import register_source
 from tram.interfaces import BaseSource

 @register_source("kafka")
 class KafkaSource(BaseSource):
     def read(self):                    # infinite generator for stream mode
         consumer = KafkaConsumer(...)
         for msg in consumer:
             yield msg.value, {"offset": msg.offset, "topic": msg.topic}

 Step 2 — Add import to connectors/init.py

 from tram.connectors.kafka import source   # one line

 Step 3 — Add Pydantic config model (optional but recommended)

 # In tram/models/pipeline.py — discriminated union entry
 class KafkaSourceConfig(BaseModel):
     type: Literal["kafka"]
     brokers: list[str]
     topic: str
     group_id: str = "tram"
     ...

 That's it. No changes to executor, registry, API, CLI, or scheduler.

 Pipeline YAML immediately supports:
 source:
   type: kafka
   brokers: [kafka:9092]
   topic: pm-raw
 schedule:
   type: stream    # runs continuously

 CLI

 Two modes:
 - Direct commands (no daemon needed): run, validate, plugins, version
 - Daemon-proxy commands (talk to running daemon via REST): pipeline *, daemon *

 # Direct (no daemon required)
 tram run <pipeline.yaml>           # Execute once, exit
   --dry-run                        # Validate only, no I/O
 tram validate <pipeline.yaml>      # Pydantic validation, exit 0/1
 tram plugins                       # Print registered plugin keys
 tram version                       # Print version

 # Daemon management
 tram daemon [--config tram.yaml]   # Start the daemon (scheduler + REST API)
   --host / --port / --log-level

 # Pipeline CRUD (proxies to running daemon's REST API)
 tram pipeline list                             # GET  /api/pipelines
 tram pipeline add <file.yaml>                  # POST /api/pipelines (upload YAML)
 tram pipeline remove <name>                    # DELETE /api/pipelines/{name}
 tram pipeline start <name>                     # POST /api/pipelines/{name}/start
 tram pipeline stop <name>                      # POST /api/pipelines/{name}/stop
 tram pipeline run <name>                       # POST /api/pipelines/{name}/run
 tram pipeline status <name>                    # GET  /api/pipelines/{name}
 tram pipeline reload                           # POST /api/pipelines/reload

 # Run history
 tram runs [--pipeline <name>] [--limit 20]     # GET  /api/runs
 tram runs <run_id>                             # GET  /api/runs/{id}

 CLI pipeline commands read TRAM_API_URL env var (default: http://localhost:8765) to find the daemon.

 12-Factor Principles

 - Config from env vars only — no hardcoded paths or credentials
 - Logs to stdout — structured JSON log output for log aggregators (Filebeat, Fluentd)
 - Stateless — state_dir is optional; run history can be in-memory (lost on restart)
 - Health endpoints — /api/health (liveness) + /api/ready (readiness) for k8s probes
 - Pipeline configs as volume mounts — no rebuild needed to add/change pipelines

 Core Abstractions

 Data Model

 - A record = dict — universal in-memory unit
 - Source yields (bytes, metadata_dict) one per file/message
 - Serializer: bytes → list[dict] (parse) and list[dict] → bytes (serialize)
 - Transforms receive and return list[dict]
 - Sink receives (bytes, metadata_dict)

 Plugin Registry

 @register_source("sftp")          # registers SFTPSource under key "sftp"
 class SFTPSource(BaseSource): ...

 source_cls = get_source("sftp")   # → SFTPSource
 source = source_cls(config)       # instantiate with config dict

 Auto-registration: connectors/__init__.py, transforms/__init__.py, serializers/__init__.py each import all submodules,
 firing decorators at startup.
