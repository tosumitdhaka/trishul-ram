# v1.3.2 Design Plan: Batch Reconciliation, Incremental Record Processing, and Safe File Output

**Status:** In progress  
**Target version:** v1.3.2  
**Branch:** observer  
**Scope:** Manager mode batch pipelines first; execution-path improvements are generic where practical.

---

## Goal

This workstream closes three gaps exposed by large PGW CDR batch runs in kind:

1. **Batch reconciliation (A)** — manager must recover from worker pod death, OOM, restart,
   or manager restart without leaving pipelines stuck in `running`.
2. **Incremental record processing (B)** — large file-based batch pipelines must process
   records in bounded memory instead of materializing full decoded + transformed + serialized
   datasets in one pass.
3. **Safe file output semantics (C)** — record-safe file outputs (`ndjson`, `csv`) must avoid
   unsafe re-append and partial final-file corruption during reruns or worker failure.

This plan is driven by a real PGW regression observed in kind and is intentionally split into
phases so runtime correctness lands before performance and output-semantics changes.

---

## Observed Failure

### Runtime symptoms

- Manager showed `pgw-cdr-ber-to-json` as `running` for hours after the worker died.
- Worker pod had no active batch run after restart.
- The batch run did not appear as a completed/failed run in `/api/runs`.
- Next scheduled batch trigger was blocked because the pipeline still appeared `running`.

### PGW test data

- Source path: `/home/dhaka/test/cdr/pgw`
- Matching files: `2`
- File sizes:
  - `vifmpgw01_20260416135355_2832` ≈ `42 MB`
  - `vigipgw01_20260416145418_53645` ≈ `21 MB`
- CSV reference row counts:
  - `FM-out.csv` = `94,276`
  - `GI-out.csv` = `47,280`
  - total = `141,556`

### Failure facts confirmed

- Worker last termination reason: `OOMKilled` (`137`)
- Previous worker log reached:
  - batch started
  - SFTP source found 2 matching files
  - one PGW output file written
  - worker then died before callback
- Existing output file was appended on rerun because:
  - source intentionally used `skip_processed: false`
  - `move_after_read` was disabled
  - sink reused deterministic filename
  - file sink defaulted to append behavior

### Root causes

1. **Manager correctness gap**
   Batch runs are not reconciled the way stream placements are reconciled.

2. **Memory pressure gap**
   The current path fully materializes:
   - source file bytes
   - ASN.1 BER record split list
   - decoded record list
   - transformed record list
   - serialized NDJSON payload

3. **Unsafe batch file output semantics**
   Re-running the same input file can append into an existing final output file.

## Non-Goals

- No checkpoint/resume of partially processed batch files in v1.3.2.
- No automatic immediate re-trigger of failed/lost batch runs by default.
- No attempt to make every serializer fully streaming in this release.
- No breaking API rename of existing placement fields or manager metrics.

---

## Decisions Locked Before Implementation

### 1. Lost batch retry policy

Default behavior:

- If a worker-owned batch run disappears, the manager marks it failed and clears `running`.
- The pipeline becomes schedulable again.
- Re-trigger happens on the **next schedule** or by manual trigger.

This avoids retry storms after deterministic failures such as OOM.

### 2. Reconciliation truth source

Manager-side batch reconciliation uses:

- manager dispatch tracking
- worker `/agent/status`
- worker assignment map

This is enough for v1.3.2. K8s termination reasons may be added later as enrichment, not as
the core reconciliation mechanism.

### 2a. Adoption timing after manager restart

Batch adoption / stale cleanup after manager restart happens in the **first batch reconciler
cycle after boot**, not inside `_boot_load()`.

Reasoning:

- `_boot_load()` already restores persisted pipeline state and stream placements.
- worker `/agent/status` probing belongs with reconciliation concerns, not DB/bootstrap wiring.
- this keeps the boot path smaller and avoids mixing worker network probes into controller
  restore logic.

Operational note:

- after a manager restart, a previously running batch may still appear `running` until the
  first reconciler tick completes.

### 3. Initial serializer scope for incremental processing

First implementation targets:

- `asn1`
- `ndjson`
- `csv`

The executor-side batching design should remain generic so other serializers can plug in later.

### 3a. Window-size config naming

The new bounded-processing window must **not** reuse existing `batch_size`, because
`batch_size` already means "stop after N input records" in the executor.

Decision:

- keep existing `batch_size` semantics unchanged
- introduce a separate bounded-processing window config named `record_chunk_size`

### 4. Output semantics for record-safe file formats

For `ndjson` and `csv`:

- incremental writes are allowed and desirable
- unsafe append-to-final-file semantics are not desirable for batch-file outputs
- rolling remains threshold-driven, not unconditional

### 5. Safe publish model

Crash-safe batch-file output should use:

- temporary/run-scoped output while a source file is being processed
- finalize/publish only after successful source-file completion

This solves retry safety better than blind append.

### 6. Reconciler identity

Batch reconciliation will be implemented as a **separate `BatchReconciler`** running alongside
the existing `PlacementReconciler`.

Reasoning:

- `PlacementReconciler` is tightly coupled to stream placement state (`slots`, `run_id_prefix`,
  placement DB status, redispatch logic).
- batch runs do not use the placement model.
- separate reconcilers keep the two state machines and test surfaces independent.

## Phase A — Batch Reconciliation

### Problem Statement

Today a worker-owned batch run can disappear without a completion callback. The manager then:

- keeps pipeline status as `running`
- does not persist a failed result for the lost run
- blocks future cron/interval fires
- requires manual intervention to recover

This is the highest-priority correctness bug.

### Target Behavior

- Manager tracks every dispatched batch run.
- Reconciler checks active batch runs every `T` seconds.
- If the tracked run is missing on the assigned worker:
  - manager synthesizes a failed run result
  - manager clears active batch ownership
  - pipeline leaves `running`
  - next scheduled fire can proceed normally
- On manager restart:
  - if pipeline is `running` but manager has no in-memory batch record, reconcile by scanning workers
  - if run is found, adopt it
  - if no run is found, mark stale batch failed

### Proposed Runtime Model

```
manager dispatches batch run
  -> controller stores active batch lease:
       pipeline_name -> {run_id, worker_url, started_at, schedule_type}

reconciler loop every T seconds
  -> inspect active batch leases
  -> query owning worker /agent/status
  -> if run_id present: keep running
  -> if run_id missing:
       synthesize failed RunResult
       manager.record_run(...)
       clear lease
       set pipeline status = error/stopped/scheduled as appropriate

manager boot with pipeline status=running
  -> _boot_load() restores pipelines only
  -> first BatchReconciler tick scans workers /agent/status by pipeline/run
  -> if found: adopt active batch lease
  -> else: synthesize stale failed run and clear running
```

## Phase B — Incremental Record Processing

### Problem Statement

Large file-based batch pipelines currently decode and serialize the entire file in one pass.
This creates peak memory proportional to the full file and its expanded decoded/output forms.

The PGW test case shows `1 GiB` is not reliable for current ASN.1 + NDJSON behavior.

### Target Behavior

- Record processing happens in bounded batches.
- Peak memory is controlled by batch size rather than total file size.
- Executor-side batching remains generic where possible.
- Serializer-specific streaming/incremental support is added where it yields real benefit.

### Design Direction

There are two layers:

1. **Generic executor batching**
   Process records in windows after parse instead of writing one huge sink payload.

2. **Serializer-specific incremental parse**
   Particularly important for ASN.1 BER split files, where parse currently returns one large
   list of decoded records.

Initial priority:

- `asn1`
- `ndjson`
- `csv`

### Serializer Interface Contract

Current serializer contract remains valid:

- `parse(data) -> list[dict]`

For incremental support, serializers may optionally implement:

- `parse_chunks(data, record_chunk_size) -> Iterator[list[dict]]`

Executor behavior:

- if serializer implements `parse_chunks(...)`, use it for bounded processing
- otherwise, fall back to existing `parse(...)`
- when fallback returns a large list, executor may slice it into `record_chunk_size` windows
  before transform/write stages

This keeps existing serializers compatible while allowing ASN.1 to avoid the full in-memory
list bottleneck.

### Threading Scope for v1.3.2

Bounded windowed processing is limited to the **serial batch path** (`thread_workers == 1`)
in v1.3.2.

The existing thread-pool batch path remains unchanged and is deferred for follow-up work.
This keeps the first implementation simpler and avoids mixing record-window semantics with
concurrent chunk submission.

Current implementation note:

- v1.3.2 currently supports bounded record batching in the serial batch path only.
- `BaseSerializer.parse_chunks(...)` falls back to eager `parse(...)` slicing for unchanged serializers.
- `asn1` implements real incremental BER decode when `split_records: true`.
- serial batch execution now writes smaller `ndjson` / `csv` payloads incrementally and can
  finalize them safely per source file when the sink supports `finalize_source(...)`.
- threaded batch execution remains on the legacy eager path in v1.3.2.

## Phase C — Safe File Output Semantics

### Problem Statement

Current file-sink behavior is too permissive for batch-file outputs:

- deterministic filenames can be reused across attempts
- append semantics can mix old and new attempts
- partial final output files survive worker death
- reruns can silently duplicate data

This is especially problematic for testing repeated reads of the same input file.

### Target Behavior

- Record-safe formats can still write incrementally.
- Final published files should not be silently corrupted by rerun append.
- Rolling remains available but should not be the only protection mechanism.
- Batch-file outputs should favor safe publish semantics over blind append.

### Design Direction

For record-safe serializers (`ndjson`, `csv`):

- allow incremental append-safe writes to a temporary/run-scoped target
- publish final filename only after successful source-file completion
- keep threshold-based rolling for large outputs

This requires a sink lifecycle hook because today the executor only calls `sink.write(...)`
per source unit and has no "source completed successfully" callback.

Decision:

- add a new optional sink lifecycle method, implemented as a no-op by default:
  - `finalize_source(meta, success: bool) -> None`
- executor calls `finalize_source(...)` once per source-file unit after all record batches for
  that source file either succeed or fail

For temp-output semantics:

- writes go to a temp/run-scoped target during processing
- `finalize_source(..., success=True)` publishes the final artifact
- `finalize_source(..., success=False)` leaves only temp/orphaned artifacts, not a published
  final file

### SFTP Finalize Semantics

Initial scope covers both local and SFTP sinks.

For SFTP:

- upload to a temp path in the target directory
- finalize with an SFTP rename into the final path
- if rename/finalize fails, treat the source-file publish as failed and leave the temp artifact
  un-published

This does not claim a cross-server atomicity guarantee; it is still preferable to direct
append into the final filename because it prevents publishing partially written final output.

Open design point to settle during implementation:

- whether default file-sink mode should remain `append` globally, or become safer for
  batch-file sinks specifically

### Checklist

- [x] Document current unsafe append-to-final-file behavior
- [x] Add optional `BaseSink.finalize_source(meta, success)` no-op hook
- [x] Define temp-output naming / run-scoped artifact model

- [x] Ensure failed runs do not publish incomplete final files
- [x] Keep threshold-based rolling for append-safe serializers
- [ ] Re-evaluate file-sink default behavior for batch pipelines
- [ ] Document operator guidance for:
  - [ ] repeat-test runs on same input
  - [ ] `skip_processed`
  - [ ] `move_after_read`

Current implementation note:

- `BaseSink.finalize_source(meta, success)` is implemented as a no-op default hook.
- local and SFTP sinks stage record-safe batch output (`ndjson`, `csv`) to run-scoped temp
  targets when `enable_safe_finalize` is set.
- the serial batch executor tracks source-file boundaries and calls `finalize_source(...)`
  once per source unit on success or failure.
- `finalize_source(..., success=False)` removes the current run temp artifact instead of
  leaving it behind.
- when a new staged write starts for the same deterministic final filename, stale
  `.tram-*.tmp` artifacts from failed prior attempts are discarded before the new run writes.
- `file_mode: single` and `file_mode: append` both use staging/finalize for these record-safe
  serializers in the serial batch path.
- threaded batch execution and stream execution do not use staged safe-finalize semantics in
  v1.3.2.

## Suggested Implementation Order

1. **Phase A** — correctness first
2. **Phase B** — bounded-memory execution for PGW/large file workloads
3. **Phase C** — safe publish/output semantics

Reasoning:

- Phase A fixes the operationally blocking stuck-run bug.
- Phase B reduces the probability of worker death under real workloads.
- Phase C hardens retry semantics and output safety once the runtime can recover correctly.

---

## Decisions Confirmed

1. **Active batch lease persistence**
   v1.3.2 uses in-memory active batch leases first, with worker-scan adoption on manager
   restart. DB-backed active lease persistence is deferred.

2. **Safe publish serializer scope**
   Safe publish semantics start with record-safe serializers only:
   - `ndjson`
   - `csv`

   Other serializers may gain serializer-specific safe publish behavior later.

3. **File-sink default behavior**
   The default-behavior review remains scoped to batch-file use cases rather than a global
   file-sink behavior change. Compatibility impact should be evaluated during Phase C before
   any default is changed.

4. **Temp-file recovery semantics**
   v1.3.2 does not implement checkpoint/resume. If temp artifacts from a failed prior attempt
   are found on rerun, they are discarded or overwritten rather than resumed.
